# CHANGES — Фаза 0.5 + Фаза 1 (старт)

## Что сделано в этой итерации

### 1. CoT-judge (`src/prompts.py`, `src/llm_client.py`, `src/filters.py`, `configs/default.yaml`)

**Проблема Фазы 0:** прямой Да/Нет judge давал 48–61% accuracy на контрфактах и не ловил числовые/датовые изменения. Run 2 ввёл регрессию на позитивах (86.7%).

**Что сделано:**
- Добавлен новый промпт `LLM_JUDGE_COT_SYSTEM`/`LLM_JUDGE_COT_USER` — структурированное 4-шаговое рассуждение с явным маркером `ВЕРДИКТ: Да|Нет` в конце.
- Парсер `parse_cot_verdict(text) → (verdict, reasoning)` в `src/llm_client.py`. Устойчив к: `**ВЕРДИКТ:**`, `вердикт — Да`, `ОТВЕТ:`, `VERDICT:`, em-dash, разному регистру, последней строке без маркера. 10 модульных кейсов — все зелёные.
- `LLMJudge` теперь mode-aware: `mode="cot"` (default) или `mode="fast"`. Старый API `judge()/judge_batch()` сохранён (используется в `DualJudge`), новый `judge_verbose()/judge_batch_verbose()` дополнительно возвращает reasoning — для отладки и отчёта.
- В конфиг добавлены `judge_mode`, `judge_cot_max_new_tokens: 320` (хватает на ~4 строки рассуждений Qwen2.5-7B).

**Стоимость:** ~300 токенов на вызов вместо ~8. На 4060 + Q4 это ~5–10 сек на judge. Для sanity (75 вызовов) — пара минут. Для скейла Фазы 4 (2500 вызовов) надо будет либо ужимать CoT, либо тыкать `mode="fast"` и опираться на CE.

### 2. fuzzy_in с русской морфологией (`src/text_utils.py`)

**Проблема:** recall 62% на ожидаемых фактах был занижен из-за substring-матчинга: `«Льва Николаевича Толстого»` (родительный) ≠ `«Лев Николаевич Толстой»` (именительный).

**Что сделано:** новый модуль `src/text_utils.py` с трёхуровневой стратегией:
1. Точная подстрока (lower).
2. Лемматизация через pymorphy3 → подпоследовательность лемм с любой из сторон.
3. Set-inclusion лемм для длинных needle (≥3 токенов) — для случаев типа `«Толстой Лев Николаевич»` ↔ `«Льва Николаевича Толстого»`.

Если pymorphy3 не установлен — graceful fallback на шаг 1 (старое поведение). `morph_available()` экспортируется для логирования.

**Sanity-скрипт** теперь импортирует `fuzzy_in` оттуда и в заголовке отчёта пишет, доступна ли лемматизация. Важно: фуззи только в `recall_vs_expected`, hallucination check остался строгим (verbatim в d⁺) — чтобы не сломать downstream-промпт для контрфактов.

### 3. Phase 1: загрузка RuBQ 2.0 (`scripts/01_load_rubq.py`)

Источники на HF: `d0rj/RuBQ_2.0` (2330 test + 580 dev) и `d0rj/RuBQ_2.0-paragraphs` (57k параграфов).

Пайплайн загрузчика:
- Загружает оба сплита вопросов + корпус параграфов.
- Индексирует параграфы по uid.
- Для каждого вопроса берёт первый параграф из `paragraphs_uids.with_answer` подходящей длины (по умолчанию 80–2500 символов).
- Длинные параграфы усекаются по последней точке.
- Бракует: пустой `with_answer`, отсутствие параграфа в корпусе, неадекватная длина вопроса.

Выходы:
- `data/rubq/rubq_full.jsonl` — все валидные пары `{qid, query, d_plus, meta}`.
- `data/rubq/rubq_dev50.jsonl` — детерминированный сэмпл 50 штук, стратифицированный по `tags` (1-hop vs multi-constraint), seed=42.
- `data/rubq/stats.json` — длины запросов/параграфов с p10/p25/p50/p75/p90, разбивка по тегам и сплитам, причины отбраковки.

В консоль печатается сводка для глаза.

## Что ожидать от прогона Фазы 0.5

Гипотезы:
- **Recall**: 62% → 75–85% (морфологические варианты теперь засчитываются).
- **Judge accuracy на контрфактах**: 48–61% → 70%+ (CoT должен ловить числовые/датовые подмены).
- **Judge accuracy на позитивах**: 86.7% (Run 2) → должна вырасти, т.к. CoT даёт модели "оправдать" документ через ШАГ 3, а не сразу резать.
- **Стоимость**: судья замедлится в ~30–50 раз (8 токенов → 300). При 75 вызовах это ~5–10 минут.

Если CoT не дал прироста на числах — стоит залезть в reasoning'и (они теперь в `outputs/sanity/judge.jsonl`) и посмотреть, что именно модель прописывает на ШАГ 3.

## Что дальше (Фаза 2)

После того как Фаза 0.5 на sanity-наборе подтвердит, что CoT-judge работает:

1. Запустить `01_load_rubq.py`, проверить, что `rubq_dev50.jsonl` собрался разумно (д_plus не слишком длинные, теги распределены).
2. Прогнать `NegativePipeline` на этих 50 примерах **БЕЗ фильтров δ** — нужно увидеть сырое распределение δ, чтобы откалибровать границы бакетов.
3. Построить гистограмму δ → выбрать границы `easy/medium/hard`.

В `configs/default.yaml` уже есть дефолтные границы (0.30/0.55/0.75/0.92), но они с потолка — реальное распределение может быть смещено.
