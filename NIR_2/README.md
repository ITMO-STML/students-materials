# NIR: Генерация обучающих данных с управляемой семантической сложностью

Прототип системы генерации обучающих данных (пар/триплетов запрос-документ)
с контролем уровня семантической сложности для дообучения моделей плотного
семантического поиска (dense retrieval).

## Идеология

Объединяем два конвейера:

1. **Counterfactual Negatives (CN)** — на основе пары `(q, d⁺)` LLM выделяет
   атомарные факты, мутирует выбранный факт так, чтобы документ перестал быть
   ответом на `q`, и тройной фильтр (δ-метрика на ансамбле энкодеров, BM25,
   кросс-энкодер + LLM-судья) отсеивает мусор и false negatives.

2. **Gradient Positives (GP)** — генерируем семейство позитивных вариантов
   `{d⁺ᵢ}` со ступенчато убывающей семантической близостью к оригиналу,
   но проходящих кросс-энкодерную проверку как валидные ответы. Используется
   как для аугментации позитивов, так и (на границе) для контролируемых
   "soft negatives".

Обе пайплайны производят размеченные по сложности данные → можно строить
куррикулум-обучение (CuSINeS-style schedule) и стратифицированное обучение.

## Структура

```
nir/
├── configs/default.yaml        # все гиперпараметры в одном месте
├── src/
│   ├── prompts.py              # все промпты (главная исследовательская субстанция)
│   ├── llm_client.py           # обёртка над Qwen (vLLM / transformers)
│   ├── encoders.py             # ансамбль энкодеров для δ
│   ├── filters.py              # δ, BM25, CE-judge, LLM-judge
│   ├── pipelines.py            # оркестрация CN и GP
│   ├── data.py                 # загрузка RuBQ / MIRACL-ru
│   └── schemas.py              # датаклассы для записей
├── scripts/run_mvp.py          # сквозной запуск на сэмпле
├── data/sample_rubq.jsonl      # 10 примеров для smoke-теста
└── outputs/                    # сюда пишем сгенерированные триплеты
```

## Запуск

```bash
# 1. Окружение (одноразово)
pip install -r requirements.txt

# 2. Smoke-тест на 10 примерах из data/sample_rubq.jsonl
python scripts/run_mvp.py --config configs/default.yaml --limit 10

# 3. Полный прогон (укажите путь к RuBQ или MIRACL-ru)
python scripts/run_mvp.py --config configs/default.yaml --data path/to/rubq.jsonl
```

## Дизайн-решения (важно для НИР)

1. **δ считается ансамблем энкодеров, а не только тем, который будем дообучать**.
   Это снижает selection bias и делает метрику сложности устойчивой к тому,
   какой ретривер мы потом обучаем.

2. **Извлечение фактов open-ended**, без зашитого словаря типов
   `[субъект, действие, результат, число]`. LLM сама описывает тип факта —
   это делает пайплайн доменно-независимым.

3. **Двойная верификация**: LLM-judge + cross-encoder. Только если оба согласны
   "это не ответ" — оставляем как hard negative. Это режет false negatives.

4. **Gradient positives — дерево, не цепочка**. Каждый `dᵢ` ветвится из `d⁺`,
   а не из `d_{i-1}`. Это устраняет семантический дрейф цепочки.

5. **Стратификация по δ** в три бакета вместо одного жёсткого диапазона
   `[0.65, 0.92]`. Это даёт более разнообразное покрытие распределения
   сложностей (см. "Beyond Hard Negatives", arXiv:2604.04734).
